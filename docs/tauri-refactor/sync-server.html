Self-Hosted Sync Topology

Each desktop install keeps its own data. The Docker server coordinates changes and blobs.

Sync Lifecycle

Local writes are immediate. Network sync is resumable and conflict-aware.

sequenceDiagram
  participant UI as React UI
  participant LOCAL as Local Rust repositories
  participant QUEUE as Sync queue
  participant CLIENT as Sync client
  participant SERVER as Docker sync API
  participant DB as Sync metadata DB
  participant BLOB as Blob store

  UI->>LOCAL: save chat/message/file/settings
  LOCAL->>QUEUE: append operation + blob refs
  CLIENT->>SERVER: push operations since cursor
  SERVER->>DB: validate and store changes
  SERVER-->>CLIENT: ack heads + missing blob refs
  CLIENT->>BLOB: upload missing blobs
  SERVER-->>CLIENT: websocket heads update
  CLIENT->>SERVER: pull missing operations
  SERVER->>DB: read changes by collection
  SERVER-->>CLIENT: changes + conflict markers
  CLIENT->>LOCAL: merge safe changes
  CLIENT-->>UI: sync://status and sync://conflict

Data Classification

Different data classes use different sync strategies. Secrets stay local unless explicitly encrypted later.

flowchart TB
  Data["Marinara local data"] --> Metadata["Metadata operations"]
  Data --> Settings["Settings"]
  Data --> Files["Blob files"]
  Data --> Secrets["Secrets"]
  Data --> LocalOnly["Local-only runtime data"]

  Metadata --> CRDT["CRDT or append-only log
chats, messages, characters, lorebooks, game notes"]
  Settings --> LWW["LWW + revision history
UI prefs, haptic, TTS, translation, generation params"]
  Files --> CAS["Content-addressed blobs
avatars, gallery, sprites, backgrounds, current profile packages"]
  Secrets --> NoSync["Do not sync by default
API keys, OAuth refresh tokens"]
  LocalOnly --> Ignore["Never sync
sidecar binaries, runtime installs, temp logs"]

Docker Compose Services

sync-api Rust + Axum

REST, WebSocket, auth, sync protocol, OpenAPI.

postgres Metadata

Devices, changes, heads, conflicts, snapshots.

object storage Local FS or S3

Avatars, gallery, sprites, generated assets, current profile packages.

redis Optional

Rate limits, queues, fanout for larger installs.
